Scalability Audit

Performed 2026-04-20 · Single Node process on one VM, Postgres 20-conn pool · Reference snapshot of the system at this point in time.

Executive summary

Current architecture ceiling: single Node.js process on one VM, backed by a single Postgres with a 20-connection pool. Practical concurrent capacity estimated:

  • Voice sessions: ~150–300 simultaneous before performance degrades
  • HTTP / portal traffic: several thousand concurrent users is fine — these are HTTP-only and cheap

To serve real "hundreds-thousands of users" the system is not blocked by a code rewrite. The bottlenecks are operational: (1) no horizontal scaling wiring, (2) single-point DB, (3) external API rate limits + cost, (4) no CDN, (5) no metrics. Each is addressable incrementally.

1. Compute model — biggest risk

What's there today

  • server.js runs as a single Node process. No clustering, no PM2, no worker threads.
  • Voice pipeline runs in the same process as the HTTP API and WebSocket server.
  • Every WS connection runs STT → LLM → TTS on the same event loop.

Why it matters

  • A single Node process tops out at ~1000 idle WS connections, but ~100–300 active voice sessions (each holds buffers, timers, hits 3 external APIs per turn).
  • One slow GPT call blocks nothing (async), but one CPU-bound op (scraping, PDF parsing, image upload) pauses the entire event loop — affecting every other user.

Headroom before action needed

Concurrent voice sessionsStatus
Up to 100FINE on one box
100–300JITTER noticeable latency, still functional
300+PROBLEMS action required

What to do

  1. Enable Node clustering (cluster module or PM2) — ~30 lines, gives N-core parallelism on same VM
  2. Move scraper + PDF parsing to a separate worker process (isolates CPU-heavy work from voice hot path)
  3. Separate WS service from HTTP service onto different VMs at ~500 concurrent sessions

2. Database — biggest silent risk

What's there today

  • pg pool with max: 20 connections (db/index.js:11)
  • 28 migration files, many indexes already in place (26 CREATE INDEX lines)
  • Zero transaction usage outside Stripe webhooks (no BEGIN/COMMIT/ROLLBACK)
  • Single Postgres instance, no read replicas

Why it matters

  • 20 connections is plenty for ~1000 concurrent HTTP users. But the pool gets exhausted fast if a handful of queries hang.
  • Zero transactions = race-condition risk in: voice-minute increments, concurrent subscription upsize+downsize, affiliate commission creation, pause/reactivate toggles.
  • No read replica = analytics queries compete with production writes.

What to do

  1. Wrap billing.js webhook handlers in transactions + idempotency keys
  2. Wrap voice-minute UPDATEs in atomic statements (SET minutes_used = minutes_used + Y)
  3. Add a read replica + route analytics endpoints to it
  4. Bump pool max to 40–60 once clustering is added

3. WebSocket capacity gates — already in place

From middleware/wsCapacity.js:

  • IP_NEW_CONN_PER_MIN = 30
  • TENANT_CONCURRENT_CAP = 100
  • IP_TENANT_CONCURRENT_CAP = 3
  • IP_DAILY_MINUTES_CAP = 120
✓ Verdict: well-designed and hardens against abuse + runaway costs. Per-tenant cap of 100 is a sane ceiling.

4. External APIs — your real scalability constraint

What's there today

  • Every voice turn hits Deepgram (STT) + OpenAI (LLM) + Cartesia (TTS) in sequence
  • No caching, no batching — every turn is fresh API calls
  • Sentry configured with tracesSampleRate: 0.1 (10% — good)

Per-user cost at scale (rough)

ProviderApprox cost
Deepgram Nova-2 STT~$0.0043 / min audio
OpenAI GPT-4o-mini~$0.001–$0.003 / turn (prompt is ~35K chars)
Cartesia Sonic-2 TTS~$0.02–$0.05 / min audio produced
Total per minute of conversation~$0.03 – $0.10

Rate limits to watch

  • OpenAI: per-account RPM (~10K RPM for gpt-4o-mini tier 2). Fine until thousands of concurrent sessions.
  • Deepgram / Cartesia: streaming concurrency limits. These are real ceilings.

What to do

  1. Monitor per-provider spend + rate-limit errors weekly
  2. Trim conversation history smarter (currently sends last 16 turns regardless)
  3. Cache compiled context (tenants.compiled_context) — already done ✓
  4. Consider a fallback model if primary fails

5. Static assets / CDN

What's there today

  • express.static serving /css, /js, /images, /client/js
  • Cache-Control: no-cache on /client/js/* (correct for dynamic JS)
  • No CDN in front of the app
  • No gzip / brotli middleware
  • No image optimization

Verdict: every static asset request hits the Node box. Fine for 1000 users — not fine for 100K.

Cheap wins

  1. Cloudflare (free plan) in front → auto CDN + DDoS + cache
  2. compression middleware → reduces payload sizes 60–80%
  3. Optimize images in /public/images (some are likely multi-MB)

6. HTTP rate limits

From middleware/rateLimit.js:

  • auth: 10 / 15min · signup: 5 / hour · api: 100 / min · scrape: 3 / 10min
  • tour-api: 60 / min · client-portal: 60 / min · affiliate: 60 / min

Issue: in-memory limiter = breaks with clustering. A user's 10 auth requests could hit 10 different workers and bypass the limit.

What to do

  1. Move to Redis-backed rate limiting (rate-limit-redis) when clustering is added

7. Memory / leaks

  • Puppeteer pool restarts browser after MAX_PAGES_BEFORE_RESTART = 50 (good — prevents leaks)
  • No explicit session cleanup code visible — relies on WS close events
  • Conversation history stored per-session in memory (not in DB for every turn)

What to do

  1. Add a monitoring dashboard with process RSS memory — alert if climbing steadily without session count rising

8. Observability — thin

  • Sentry initialized ✓ (good for errors)
  • Abundant console.log / console.error (good for tracing, overwhelming at scale)
  • No metrics system (no Prometheus, no Datadog, no StatsD)
  • No structured logging, no distributed tracing

What to do

  1. Add a basic metrics dashboard (Grafana Cloud or OTel + Honeycomb free tier)
  2. Track: voice session count, per-provider latency, DB query time p50/p95/p99, error rate, cost per session

9. Deployment / backup

  • Backup Scheduler started (daily) — but content + restore procedure unverified
  • KeepWarm pinger every 4 min — implies serverless-adjacent host that sleeps idle processes

What to do

  1. Verify daily backup actually works and test a restore once
  2. Document the restore procedure
  3. Add a staging environment (separate DB) if not already

10. Security at scale

  • ✓ JWT auth middleware
  • ✓ Rate limiting
  • ✓ Stripe webhook signature verification
  • ✓ express.json({ limit: "15mb" }) body size capped
  • ✓ WS abuse gates
  • ✗ No helmet middleware (no HSTS, no CSP, no X-Frame-Options)
  • ! Cookie-based session auth = some endpoints exposed to CSRF
  • ✓ SQL injection: all queries use parameterized $1, $2

What to do

  1. Add helmet middleware — 2 lines, big security hygiene win
  2. CSRF protection for cookie-auth POST endpoints

Priority actions

For <100 users today → 1,000 users in 3–6 months

  1. Cloudflare in front (1 hour, free) — CDN + DDoS
  2. Add helmet + compression middleware (20 min)
  3. Enable Node clustering + switch rate limiter to Redis (1 day)
  4. Verify backups + test restore (2 hours)
  5. Wrap billing webhooks in transactions with idempotency (½ day)
  6. Add a basic metrics dashboard (½ day)

For 10,000+ users

  1. Separate voice WS service from HTTP service
  2. Postgres read replica
  3. Move scraper to a dedicated worker
  4. Move puppeteer to a queue-based system (BullMQ + Redis)
  5. Move to proper Kubernetes / autoscaling group

What the system does well already

  • OK Clean middleware layering
  • OK Rate limiting at every sensitive endpoint
  • OK WS abuse protection (tenant caps, IP caps, daily minute caps)
  • OK Puppeteer memory-leak protection via restart threshold
  • OK Parameterized SQL everywhere (no injection risk)
  • OK Stripe webhook signature verification
  • OK Sentry already wired
  • OK Per-tenant volume pricing capacity gates
  • OK 28 migrations cleanly tracked
  • OK Admin-role bypass + DISABLE_AGENT_CAP env var for flexibility
Bottom line: the system is in surprisingly good shape for a solo-built platform. The ceiling is lower than it needs to be because of operational gaps (no clustering, no CDN, no metrics) — not architectural ones. None of this requires a rewrite — it's a 1–2 week hardening pass when justified by paying customers.
